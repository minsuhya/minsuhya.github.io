# AI 코드 품질 게이트 — 팀 온보딩 가이드

> 이 게이트 시스템을 사용하는 레포지토리에 합류한 엔지니어를 위한 1페이지 오리엔테이션

---

## 왜 이 게이트가 존재하는가

AI 코딩 도구는 PR 처리량을 2–4배 늘려줍니다. 그것이 이점입니다.  
위험은, 동일한 도구가 로직 오류·컨텍스트 불일치·보안 엣지 케이스를 잡아내는 인간의 판단력을 우회한다는 것입니다 — 코드가 *올바르게 보이기* 때문입니다. 실제로 그렇지 않더라도.

이 게이트는 두 효과를 분리하기 위해 존재합니다:  
**AI 도구는 빠른 개발자를 더 빠르게 만듭니다. 게이트는 약한 코드가 더 약해지는 것을 막습니다.**

게이트 없이는 일반적으로 이런 일이 벌어집니다:
- "LGTM" 러버덕 리뷰 (읽지도 않고 승인)
- 린트는 통과하지만 비즈니스 로직 오류로 프로덕션에서 터지는 AI 코드
- 지친 리뷰어에게 관용적 코드처럼 보이는 보안 취약점

---

## 3단계 워크플로우

모든 PR은 세 개의 레이어를 자동으로 통과합니다:

**레이어 1 — 정적 분석** (약 2분, 완전 자동화)  
Gitleaks가 하드코딩된 시크릿을 탐지합니다. Bandit/Semgrep이 알려진 취약점 패턴을 탐지합니다 — 프로덕션 코드와 테스트 코드 모두 스캔됩니다. Radon이 순환 복잡도 20 초과 함수에 플래그를 답니다. 커버리지 임계값으로 테스트가 생략되지 않았는지 확인합니다.  
→ *CRITICAL 발견 시 하드 블록. 나머지는 소프트 경고.*

**레이어 2 — LLM 사전 리뷰** (약 1분, 완전 자동화)  
교차 모델 LLM이 의미론적으로 diff를 분석합니다 — 로직 오류, 보안 패턴, N+1 쿼리, 아키텍처 불일치. 결과가 구조화된 PR 코멘트로 게시됩니다.  
→ *CRITICAL 시 하드 블록. HIGH 시 시니어 에스컬레이션.*

**레이어 3 — 인간 리뷰** (당신)  
PR 템플릿의 25항목 체크리스트를 적용합니다. 자동화가 잡지 못하는 것에 집중합니다: 비즈니스 로직 정합성, 전체 코드베이스 컨텍스트에서의 아키텍처 적합성, 유지보수성.  
→ *당신의 판단이 최종 게이트입니다. 중요한 것에 사용하세요.*

---

## 핵심 원칙 하나

> **게이트는 블로커가 아니라 신호입니다.**

빠른 팀은 신호를 더 빠르게 활용합니다. 레이어 1이나 레이어 2가 무언가를 플래그할 때, 당신의 역할은 그것을 무시하는 것이 아닙니다 — 빠르게 확인하고 수정하거나, 왜 거짓 양성인지 PR 코멘트에 문서화하는 것입니다.

게이트 때문에 속도가 느려진다면, 우회하는 것이 답이 아닙니다 — 규칙을 개선하는 것이 답입니다. 이슈를 여세요.

---

## 첫 번째 PR에서 해야 할 것

1. `[x] 이 PR에는 AI 생성 코드가 포함되어 있습니다` 체크박스를 정직하게 체크하세요 — 룰셋 보정에 도움이 됩니다.
2. diff를 열기 전에 레이어 2 코멘트를 읽으세요. 어디에 집중해야 할지 30초 브리핑을 줍니다.
3. 체크리스트를 순서대로 진행하세요. CI가 이미 통과했다면 `[자동]` 표시 항목은 건너뛰어도 됩니다.
4. 레이어 2 발견 사항에 동의하지 않는다면, 그냥 무시하지 말고 리뷰 코멘트에 한 줄 설명을 추가하세요.

## 자주 묻는 질문

**"배포 속도가 느려지지 않나요?"**  
레이어 1과 2는 실시간으로 약 3분이 추가됩니다. 레이어 3은 체크리스트가 내재화될수록 빨라집니다. 3주 차부터 프로덕션 인시던트가 줄어들면서 전체 배포 속도는 빨라집니다.

**"CI가 깨져있으면 어떻게 하나요?"**  
`@[팀리드]`를 태그하고 이슈를 여세요. 리드가 명시적으로 승인하지 않는 한 게이트 없이 머지하지 마세요 — 그리고 문서화하세요.

**"LLM이 문제없는 것을 플래그했어요 — 어떻게 하나요?"**  
봇 코멘트에 한 줄 설명으로 답변하세요: `"거짓 양성: 이 패턴은 [이유] 때문에 의도적입니다"`. 감사 추적이 생기고 시간이 지나면서 프롬프트 개선에 도움이 됩니다.

**"PR이 500줄이 넘어서 레이어 2가 스킵됐어요 — 이제 어떻게 하나요?"**  
가능하면 PR을 분할하세요. 불가능하다면 도메인 오너가 직접 리뷰하도록 하세요. 500줄 초과 PR은 AI 사용 여부와 무관하게 통계적으로 결함률이 더 높습니다.
